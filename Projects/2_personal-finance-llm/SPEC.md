# Specification: Finance AI (Privacy-First Desktop Application)

## 1. Vision & Core Principles
*   **Privacy-First**: No raw financial data (merchants, exact dates, transaction IDs) should ever leave the user's local machine. 
*   **Local Knowledge First**: The application must process data locally using a "Sidecar" intelligence layer (MCP) to provide summaries to the remote LLM, ensuring low latency and high security.
*   **High Density / SaaS Aesthetic**: The interface must follow modern SaaS standards (Linear/Stripe), prioritizing information density and structural stability over decorative white space.

## 2. Architecture
*   **Host**: Electron (Node.js Main Process + React/TypeScript Renderer).
*   **Data Layer**: Local SQLite database (`finance.db`) containing transactions, categories, accounts, and income records.
*   **Intelligence Sidecar (MCP Server)**: A Node or Python-based Model Context Protocol server that handles heavy data lifting, SQL execution, and local summarization.
*   **Brain**: Remote LLM (Claude 3.7+ via OpenRouter) used only for high-level reasoning and final synthesis.

## 3. Page Specifications

### Page 1: Home (Financial Command Center)
*   **Goal**: Provide an instant, high-level pulse of financial health.
*   **Core Metrics**: 
    *   Savings Rate (Income vs. Spending ratio).
    *   Financial Runway (Estimated months of survival based on liquid cash and average burn).
    *   Net Cashflow (Current month's surplus/deficit).
*   **Daily Insights**: A dynamic feed of 3-5 high-level observations generated by the local intelligence layer (e.g., spending anomalies, goal progress).
*   **Pinned Insights**: A dedicated area to display LLM-generated charts that the user has "pinned" from previous chat sessions for longitudinal tracking.
*   **Deep Dive CTA**: Every metric and insight must have a "Deep Dive" action that transitions the user to the Chat page with the relevant context pre-loaded.

### Page 2: Transactions (Global Search)
*   **Goal**: Provide a lightning-fast, searchable ledger of all historical data.
*   **Functionality**:
    *   Unified search bar scanning across Description, Category, and Account.
    *   **Data Integrity**: Must exclude internal transfers (`is_transfer=0`) by default to show actual income/outgoings.
    *   **Performance**: Must pre-fetch the most recent 100 transactions in the background so the view appears instantaneous.
*   **UI Structure**: A stable, fixed-layout table that does not jitter or shift when search results update.

### Page 3: AI Chat (Expert Analysis)
*   **Goal**: Provide free-form, conversational deep dives into complex financial questions.
*   **Interaction Model**: 
    *   **Headless Mode**: The LLM is strictly forbidden from conversational filler until all tool data is collected.
    *   **Context Pruning**: Only the last 10 messages of history are sent to the cloud to maintain speed and efficiency.
    *   **Thinking State**: Technical details (SQL logs) are hidden behind a single, elegant progress indicator.
*   **Visuals**: Charts generated in chat must be fully responsive, high-resolution (DPI 100+), and never cut off labels or legends.

## 4. MCP Server & Performance Mandates

### Local Aggregator Pattern
The MCP server must provide high-level tools that return **Markdown Summaries** rather than raw JSON.
*   `get_financial_context`: Returns a text-based summary of categories and trends.
*   `generate_standard_chart`: Executes pre-baked local Python templates for common charts (Cashflow, Spending, Merchants).

### Latency Targets
*   **Initial Intent**: < 2.5 seconds.
*   **Data-Augmented Response**: < 8.0 seconds total (including local SQL and remote reasoning).
*   **Chart Rendering**: < 5.0 seconds using the standardized fast-path.

## 5. Agentic Implementation Protocol (Opus 4.6 Workflow)

When implementing or modifying this app, use the following team structure:

1.  **Architect**: 
    *   Analyzes requests against this SPEC.
    *   Checks the `debug/*.json` logs to identify round-trip bottlenecks.
    *   Designs the local SQL queries needed to minimize remote context.
2.  **Implementer**: 
    *   Writes surgical code changes.
    *   Must ensure all UI components take 100% height/width of their parents to prevent "empty page" bugs.
    *   Must use `bbox_inches='tight'` for all Python chart generation.
3.  **UI Challenger**: 
    *   Verifies that any new UI follows the High-Density SaaS mandate.
    *   Checks for consistent font sizes (standardizing on 14px/15px Inter).
    *   Flags excessive white space or non-proportional layouts.
4.  **Technical Tester**: 
    *   Runs the full Vitest suite (`npm test`).
    *   Performs DOM integrity checks to ensure containers haven't collapsed.
    *   Validates that the `localIntentRouter` correctly catches new prompt variations.

## 6. Constraints
*   **No UI Bloat**: Do not use massive utility-first frameworks if the build pipeline is not explicitly verified. Prefer Standard CSS with CSS Variables for robustness.
*   **Strict Truncation**: Hard truncate any local tool output at 4,000 characters before sending to the LLM.
*   **No Redundant Round-trips**: If data can be fetched locally, it *must* be fetched before the first API call.
