## 14:30 — LLM Infrastructure Deep Dive for Finance App

**What was done**:
- Deep Q&A on Cursor AI's business model (wholesale API pooling, rate limiting via proxy, local models on own infra)
- Explored self-hosting options for open-source models
- Discussed embedding local models in Electron apps (LM Studio, node-llama-cpp, Ollama sidecar)
- Evaluated Qwen3.5 models (newly released Mar 2, 2026) for potential use in finance app
- Discussed multi-model routing / intent classification patterns
- Compared OpenRouter vs Together AI pricing

**What was learned**:
- Cursor proxies all requests through their gateway for rate limiting/tracking; adds ~5-20ms latency but negligible vs LLM inference time
- Cursor's "Privacy Mode" = contractual zero-retention + no training data; enterprise gets BAA. Finance app's privacy pitch is stronger: DB stays local, LLM only sees query results not raw transactions
- LM Studio is an Electron app with llama.cpp bundled, provides OpenAI-compatible API at localhost:1234. Not an embeddable SDK — it's a UX wrapper for end users.
- node-llama-cpp is the actual npm package for embedding llama.cpp natively in Electron
- Qwen3.5 small series released March 2, 2026: 0.8B, 2B, 4B, 9B. Apache 2.0. The 9B reportedly beats GPT-4o-mini and can run on 16GB Mac.
- 7B Q4 model = 4.1GB file, needs ~6GB RAM. Unworkable on 8GB MacBook Air M1 (OS takes 3-4GB already).
- Together AI: open-source models from $0.10/M tokens. Llama 3.1 8B, Qwen models available. Cheaper than OpenRouter for OSS (OpenRouter adds ~5% fee).
- Claude 3.7 Sonnet on OpenRouter: $3/M input, $15/M output
- The finance app's localIntentRouter (line 367, finance-core.ts) already does keyword-based routing — upgrading to ML-based routing (RouteLLM, Semantic Router) would be next level but current keyword approach already works well
- 0.5B-0.8B models: only viable for classification/routing, not SQL generation with multi-table JOINs
- Multi-model routing saves 60-70% on API costs by routing 70%+ of simple queries to cheaper models

**What was decided**:
- Embedded 7B model in app is NOT the right path (too heavy for low-spec users, huge download)
- Optional Ollama sidecar is the pragmatic middle ground for local/offline mode
- Worth testing Qwen3.5-9B for SQL generation to potentially replace Sonnet on simple queries

**Open threads**:
- Test Qwen3.5-9B SQL generation quality vs Claude Sonnet on the finance app's actual query patterns
- Implement multi-model routing: Together AI for simple queries, Sonnet for complex ones
- Evaluate Together AI as direct integration (skip OpenRouter for OSS models)

**Files touched**:
- None (conversation only)

**Retain**:
- Qwen3.5 small series released March 2, 2026 (0.8B–9B); 9B beats GPT-4o-mini, runs on 16GB Mac, Apache 2.0 — prime candidate to replace Claude Sonnet for simple finance queries at $0.10/M vs $15/M.
- Finance app privacy pitch: SQLite DB stays local on user's machine; LLM only sees query results, not raw transactions — stronger privacy guarantee than Cursor's privacy mode.
- Together AI for open-source models is ~5% cheaper than OpenRouter for same models; for Claude you must use OpenRouter/Anthropic directly.
- Embedding a 7B model in the Electron app requires 4.1GB download + 6GB RAM — not viable for general consumers; use optional Ollama sidecar pattern instead.
- Multi-model routing pattern: route ~70% of simple lookups to cheap 8B model (~$0.10/M), reserve Sonnet ($3/$15/M) for complex multi-step SQL/reasoning queries.

## 20:59 — Demo App Chart Fix, Latency Analysis, Slash Command Registration

**What was done**:
- Fixed broken chart rendering in demo-app: replaced `file://` URL with base64 data URL (`data:image/png;base64,...`) — Electron CSP blocks `file://` in renderer
- Analyzed 21s latency: 98% is 3 sequential OpenRouter API calls (3.5s + 7.7s + 9.4s). SQL/tools only ~400ms total
- Removed `node_modules/` from git history using `git filter-repo` — 159MB Electron binary was blocking GitHub push
- Created 15 symlinks from `.claude/commands/` → `.claude/skills/` to make skills appear in `/` autocomplete
- Committed and pushed changes to `origin/main`

**What was learned**:
- Electron's Content Security Policy blocks `file://` URLs in the renderer process. Fix: read PNG as Buffer in main process, convert to base64, send as data URL
- Claude Code has two separate slash command registries: `.claude/commands/*.md` for `/` autocomplete; `.claude/skills/<name>/SKILL.md` for AI-invocable Skill tool. Need symlinks to bridge.
- `git rm -r --cached` only removes from current tracking, NOT from history. Must use `git filter-repo` to rewrite history for large blobs.
- `git filter-repo` removes the remote after rewriting — must re-add and force push
- OpenRouter API latency for Claude Sonnet: 3.5–9.4s per call. Number of API round trips is the dominant factor, not tool execution time.

**What was decided**:
- Use base64 data URLs for all chart rendering in Electron (not file:// or custom protocols)
- Symlink approach for commands→skills to avoid content duplication
- Latency optimization: reduce API round trips by removing "always run dashboard" instruction
