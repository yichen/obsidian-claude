## 09:34 — CMA Activity Parser + DB Backup Strategy

**What was done**:
- Implemented CMA transaction parser in `ingest_fidelity_accounts.py`: `parse_cma_activity()` + `categorize_cma_transaction()`
- Parses 5 activity section types from Fidelity Z26-474983 (primary checking): Deposits, Withdrawals, Checking Activity, Exchanges In, Exchanges Out
- Built categorization rules for ~30 payee patterns with priority system: Sheri Martin (Capital One ******0615) → amount-based split, OFW → Kids Cost, plus CC payments, mortgage, insurance, transfers, etc.
- Added `fidelity_cma_transactions` table to `finance_db.py` with import logic
- Regenerated all 13 YAMLs (Jan 2025 – Jan 2026): 481 transactions, 0 uncategorized, all existing validations still pass
- Added backup/restore system: `backup-rules` exports 751 categorization rules to `Finance/categorization_rules.json`, `restore-rules` reimports them. Auto-export on every `backup_db()` call.
- Updated `Finance/CLAUDE.md` with CMA transaction docs, backup/recovery section, and full rebuild procedure

**What was learned**:
- Fidelity CMA Activity sections are within per-account page groups, not standalone pages. Must use `split_by_account()` output.
- Multi-line payee entries (e.g., "SHERI MARTIN / CAPITAL ONE N.A. ******0615") span 2-3 lines after the main transaction line. Continuation detection: no MM/DD date prefix.
- Page breaks inject noise into continuation lines: "INVESTMENT REPORT", "Account #", "YI CHEN", single-letter artifacts (`S`), 8+ digit barcode numbers, underscore strings (`BBBBB_FWLBBBBHTDJRB_`), "EC_RM", month names. All must be filtered.
- "Cards, Checking & Bill Payments" in Account Summary is a sub-total category, NOT a separate activity section. The actual items (SAMMPLAT bill pay, checks) appear in Withdrawals and Checking Activity sections respectively.
- Checking Activity uses format `CheckNumber MM/DD Code Description Amount` — different from Deposits/Withdrawals format.
- Exchanges In/Out use securities format `MM/DD SecurityName CUSIP Description Quantity Price Amount` — only need date + description + amount.
- "Insufficient Funds" entries appear as deposits with negative amounts (reversal of failed EFT). Correctly categorized as Transfer In (Reversal) with is_transfer=1.
- Sheri Martin payment classification by amount works well: $1,254 exact = Child Support, $50K exact = Property Settlement, >= $3K = Spousal Maintenance. The few "Divorce — Other" items ($960, $1,895, $2,848) are irregular settlement-related transfers around the May 2025 finalization.
- "Apa treas" = Airbnb rental income (short-term rental payout), "Solium Inc" = RSU/ESPP proceeds (Salesforce equity), "4 Corners Fina" = legal/financial services vendor.
- The finance DB's only irreplaceable data is the 751 categorization rules. All transaction data can be reimported from source CSVs/YAMLs. The rules were built up over many interactive `/finance review` sessions — only ~230 are in the Python seed data.

**What was decided**:
- Auto-export categorization rules JSON on every import (part of `backup_db()`) — no manual step needed
- JSON file syncs via Obsidian Sync, so rules survive even if the DB is deleted or machine is lost
- Full rebuild is now documented as an 8-step procedure in Finance/CLAUDE.md

## 08:15 — 2025 Spending Report, Fidelity Card Merge, Balance Validation

**What was done**:
- Generated full 2025 CC spending report by category ($176K total, $14.7K/mo across 7→6 cards)
- Categorized all 33 uncategorized BofA Atmos transactions (18 new rules created)
- Created 3 new shopping subcategories: Camping & Outdoor, Photography, Office — for hobby/work spend tracking
- Created Fees & Interest top-level category for annual fees and late fees
- Discovered fidelity-rewards and fidelity-credit-card were duplicate cards (450 identical transactions)
- Merged fidelity-rewards into fidelity-credit-card: deleted 450 dupes, removed card from ingestion pipeline and seed data
- Validated merge with CSV-vs-DB check (52 files, 0 mismatches) and PDF balance chain check (37 transitions, 0 breaks)
- Built balance validation system: `extract_balance_summary()` on all 4 parser types (Chase, Fidelity, BofA, Apple Card)
- Added 3-tier validation: (1) inline during PDF parse, (2) `verify_all_balances()` at end of `run_ingest()`, (3) `finance_db.py validate` post-DB-import
- 153/154 statements pass; 1 Apple Card $13.24 diff (device tax), 1 Apple Card chain break (Apr 2024 parse failure)

**What was learned**:
- Credit card balance formula: `new_balance = previous_balance - sum(csv_txns)` — initially used + and all checks failed; payments (positive in CSV) REDUCE balance, charges (negative) INCREASE it
- Apple Card Monthly Installments (device payment plans) affect the statement balance but are NOT in the regular transaction list — must be extracted separately and added to the balance equation
- Fidelity had two card names (fidelity-rewards, fidelity-credit-card) for the SAME physical card — the rewards card had a subset of the credit card's statements. All 450 rewards transactions had exact matches in the credit card.
- BofA "BA ELECTRONIC PAYMENT" = card payment (should be is_transfer=1, not uncategorized)
- Annual fees and late fees are real costs (not transfers) — recommended creating Fees & Interest category rather than flagging as transfers
- B&H Photo purchases span multiple categories — can't use a single keyword rule; need per-transaction categorization based on amount/date matching to order history
- Tesla Reservation ($801) is a car purchase deposit that becomes part of total purchase price

**What was decided**:
- fidelity-rewards removed permanently from pipeline — fidelity-credit-card is the canonical source
- Fees & Interest is a top-level category (not a child of Transfers)
- REI goes under Camping & Outdoor (primarily outdoor retailer even though some items are clothing)
- Balance validation runs automatically at end of every `/ingest-cc` run and after every `finance_db.py import`
- $3 threshold for balance mismatches (allows PDF text extraction rounding)

**Open threads**:
- Apple Card Apr 2024 balance extraction fails (parser doesn't match older format) — minor, only affects chain check
- Apple Card Oct 2024 has $13.24 unexplained diff — likely device tax not captured in installment line
- B&H Photo individual transactions need manual categorization (monitors → Office, camera → Photography) since one merchant spans categories
