## Put Boxes Into the Warehouse I (1564) — Medium
**Topics**: Greedy, Sorting

**Approach**:
Recognized it as a greedy problem. Sort boxes and process warehouse rooms left to right, tracking the effective height (minimum height seen so far) as you go deeper into the warehouse.

**Learnings**:
- When preprocessing warehouse rooms, a room that's too tall doesn't get removed — it just gets its effective height capped to the last known lowest height seen so far
- The warehouse height constraint is cumulative: a tall room behind a short room can only accept boxes up to the short room's height

**Mistakes / Edge Cases**:
- Initially used a stack intuition for building the candidate room list, then when converting to a list-based approach, incorrectly discarded rooms that were taller than a previous room
- The fix: keep all rooms but modify the height of taller rooms to the running minimum — a tall room still holds boxes, just not to its original height

## Accounts Merge (721) — Medium
**Topics**: Union-Find, Graph, DFS

**Approach**:
Tried to solve by merging account lists directly. Didn't initially recognize this as a Union-Find problem — only realized after discussing with Gemini.

**Learnings**:
- This is a classic Union-Find problem: emails are nodes, and shared emails between accounts create edges that union the accounts together
- Read the problem carefully — the question asks for deduplicated emails in sorted order, easy to miss both requirements
- When copying a list into a result, always clone it instead of adding a reference. Same pitfall as in recursive algorithms — mutating the original list later will corrupt the result

**Mistakes / Edge Cases**:
- Didn't handle duplicate emails within the same account's input — assumed inputs were already unique
- Missed the requirement to return emails in sorted order
- Added list references instead of clones to the result, causing mutation bugs
- Didn't recognize Union-Find as the intended approach — tried to make a direct merge work instead
