# **Spark 3.5.6 Upgrade Risk Assessment for EMR 7**

Status: *Under review*  Author: [Yi Chen](mailto:yichen@stripe.com)    Created: Aug 30, 2025  Last Updated: Aug 30, 2025

# Goals

As part of the EMR 7 migration, we need to assess the impact of upgrading the Spark runtime from EMR 6.11.1 (Spark 3.3.x) to EMR 7.10.0 (Spark 3.5.6). The purpose of this document is to provide enough context for a teammate to take ownership of the risk assessment and carry it forward into a concrete migration plan.

Specifically, this risk assessment should:

* **Identify known risks** introduced by the Spark and runtime upgrades (e.g., Scala/Python version changes, API deprecations, SQL/function behavior changes, structured streaming differences, JDBC pushdown defaults, YARN config deprecations).  
* **Evaluate the impact** of these risks on our existing workloads (Scala jobs like DIG, PySpark jobs like Merlin/SQLPen, Hudson notebooks, streaming pipelines).  
* **Document mitigations and compatibility configs** that preserve functionality during transition, and highlight where code or build changes will be required.  
* **Classify risks by severity and ownership** so we can prioritize testing and fixes.  
* **Produce a clear set of open questions** that need to be answered before we start pilot migrations.

The exit state of this work is a **concrete migration readiness plan**: we know which jobs/configs need updates, which risks are mitigated, and what remains to be tested. That enables us to move confidently into the next phase—pilot migrations—where we validate assumptions and uncover any unknown risks in production-like environments.

# Example Potential Risk Area

## Java version upgrade

* Addressed separately here:[Java & Python Upgrade for EMR 7](https://docs.google.com/document/d/19chWBBas9NwTk9lOUl09-BM5g3nZqUFxBVtDhWyj7v4/edit?tab=t.0)  
* EMR 7 uses spark-log4j2. Any use of spark-log4j will fail. Looks like we just need to [clean up](https://hound.tinyspeck.com/?q=spark-log4j&i=nope&literal=nope&files=&excludeFiles=&repos=)?

## PySpark-Specific Risks

* Python version from 3.7 to 3.11. How does this affect our PySpark jobs (Merlin/SQLPen)?   
* Question for PySpark users (Merlin/SQLPen): Is there any concern for Pandas/PyArrow versions?

## Parquet/ORC changes in Spark 3.4

* Since Spark 3.4, vectorized readers for nested types are enabled by default. This can potentially impact jobs running on EMR 6.11.1 (with Spark 3.3). 

## JDBC (DataSource V2) pushdowns default changes in 3.5

* Spark 3.5 turns several JDBC pushdowns on by default. Ideally we should use the new default.  
* How can this impact our infrastructure? 

## Structured Streaming / Kafka streaming

* Review ACLs/semantics and keep/flip useDeprecatedOffsetFetching accordingly. How would this impact our workload?   
* Trigger.Once depreciated from Spark 3.4. Do we use this in our code?  
* Default for spark.sql.streaming.kafka.useDeprecatedOffsetFetching flipped to **false** (newer offset path). The impact can be permission errors / offset fetch failures

SparkSQL changes

* Breaking change: Spark 3.5 makes to\_binary throw on malformed input. Scan code for to\_binary to see if this impact our workload.   
* ANSI SQL: in Spark 3.5 many unreasonable implicit casts (string-\>int, double-\>boolean) now error instead of silently coercing. So the jobs worked in Spark 3.3 may fail in Spark 3.5.6

Scala API risks

* Breaking Change: Using `org.apache.spark.sql.functions.udf(AnyRef, DataType)` is not allowed by default. Remove the return type parameter or set   
* Impact: custom UDFs in Scala jobs will need modification.  
* Look at our code base, find potential impacted users, assess the risk and come up with a plan

YARN configs 

* `spark.yarn.max.executor.failures` are deprecated. See [our usage](https://hound.tinyspeck.com/?q=spark.yarn.max.executor.failures&i=nope&literal=nope&files=&excludeFiles=&repos=). 

Notebook Integration

* How are Hudson jobs impacted?   
* Do the Notebooks need to be upgraded?  
* What need to be done?

# Exit Criteria

The risk assessment is complete when:

* **All known questions are addressed** with documented evidence; any unanswered items are explicitly captured as **open questions**.  
* **Key risks are documented** with clear **severity classification** (High / Medium / Low) and associated list of **impacted users, jobs, or systems**.  
* **Required DI team tasks** (build system changes, config updates, CI/CD adjustments, etc.) are identified and scoped with estimated effort, so they can be planned ahead of migration.  
* **Mitigation strategies** are defined for each discovered risk, including whether the mitigation is immediate (config change), medium-term (code refactor), or long-term (deprecation clean-up).  
* The output provides a **readiness view**: what is safe to migrate as-is, what needs changes first, and what must be validated during pilot migrations.
